# Завдання 5: Моделювання витоку пам’яті при помилці системного виклику

## 🎯 Мета роботи
Реалізувати сценарій, у якому **витік пам’яті (Memory leakage)** — тобто ситуація, коли виділена пам’ять не звільняється — виникає виключно при помилці системного виклику та залишається непомітним під час стабільної роботи програми.

---

## 💻 Опис реалізації
Програма імітує роботу з динамічним масивом даних:
1. **Перший етап**: Успішне виділення 1 КБ пам'яті через `malloc`.
2. **Другий етап (Успіх)**: Успішне розширення блоку до 2 КБ через `realloc`. У цьому сценарії програма працює коректно, і вказівник оновлюється.
3. **Третій етап (Помилка)**: Спроба розширити блок до критичного розміру `(size_t)-1`. Це призводить до передачі в `realloc` аномального значення, яке сприймається як від'ємне або надто велике.

### Причина витоку
Коли системний виклик `realloc` повертає `NULL`, програма негайно припиняє виконання (обробка помилки), але не викликає функцію `free()` для початкового блоку пам'яті. Таким чином, виділені раніше 2048 байт стають недосяжними, але залишаються зайнятими в купі.

---

## 📊 Аналіз результатів (Valgrind)
Для верифікації витоку використано утиліту **Valgrind**. Результати тестування на скриншоті підтверджують теорію:

* **Діагностика**: Valgrind виявив підозрілий аргумент `size: -1` при виклику `realloc`.
* **Статус витоку**: У звіті вказано **"definitely lost: 2,048 bytes in 1 blocks"**.
* **Heap Summary**: Зафіксовано, що під час завершення процесу (at exit) у використанні залишилося 2,048 байт.
